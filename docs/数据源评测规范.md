# 数据源评测规范

本文档定义 ani-subs Web (CSS selector) 数据源的**端到端评测标准**:怎么测、指标怎么算、广告怎么判、线路怎么分级、报告长什么样。

- **权威实现**: `.agents/skills/datasource-eval/`(agent skill,含 `scripts/run_eval.py`、`deep_sample.py`、`gen_report.py`)。报告由 `gen_report.py` **确定性生成**,不手写。
- **修改本规范时**,必须同步修改 skill 的 `SKILL.md` 与脚本实现;反之亦然。
- 评测产物在 `reports/<YYYYMMDD>-css-eval/`。
- 注意:本文的能力分级 **T0–T6** 与 `subs/web/t0/`–`t4/` 的**订阅目录分层是两回事**,后者是客户端订阅里源的静态优先级,前者是评测得出的线路实测能力。

## 1. 评测方法

跨 **3–5 部近年完结的热门番剧**(各源收录率高,能区分"源失效"与"没收录"),对每个 Web selector 源:

1. **全流程解析**: `searchSubjects → selectSubjects → searchEpisodes → selectEpisodes → selectMedia → extractVideo`,`all_channels` 模式,**每条线路都解析**(不设条数上限)。
2. **全量实播 + 基础指标**: 每条 resolved 线路都用 Animeko 桌面播放器(**mpv**,mediamp 内核,native 库自动加载)真实播放,测**可播性与起播耗时**;分辨率、码率、编码用 **ffprobe/ffmpeg 实测**(流信息 + `-c copy` 拷贝采样算平均码率,见 §2)。
3. **深度采样**: 对可用源(跨番至少成功 1 次)按**每部番 × 每条线路全覆盖**。每条线路先用 `detect_hls_ads` 做 **HLS 结构预筛**(拉 m3u8 跑结构启发式 + Ani 真实客户端广告过滤器 HlsManifestFilter,见 §3),疑似插入广告段的**中点(≤55s)自动加为截图点**并相应延长播放(基础 28s,上限 60s);长播中在 0/3/8/15/25s + 加采点各截一帧。
4. **广告判定**: agent **逐张真实读图**(见 §3),覆盖全部实播成功过的线路;结构预筛的疑似时间段作为看图提示与"滤不滤得掉"依据。
5. **生成报告**: `gen_report.py` 按本规范产出三层报告(总索引 / 源页 / 线路页)。

要求**完整**:每个源 × 每部番 × 每条线路都要真实播放实测,不许抽样。

## 2. 指标口径

指标全部**实测**,不是源声称的清晰度:分辨率/码率/编码**优先取 ffprobe/ffmpeg 实测**,起播/可播性取 **mpv 真实播放**(快测 4s + 深采长播两路数据合并)。

| 指标 | 单线路聚合口径 | 来源(优先级从高到低) |
|---|---|---|
| 分辨率 | 跨番实测的**众数** | ffprobe 流信息 `width x height`;缺则 mpv `mediaAnalysis.video` |
| 码率 | 跨番实测的**中位数** | ① ffmpeg `-c copy` 拷贝采样实测均值 `size*8/duration`(`bitrateSource=ffprobe_measured`);② ffprobe `format.bit_rate`(`ffprobe_format`);③ 播放器统计 `overallBitrate`/视频流 `bitrate`(`player`)。HLS 直连的 `format.bit_rate` 常缺/不可靠,故以拷贝实测为准,报告标注码率来源 |
| 编码 | 跨番实测的**众数** | mpv `mediaAnalysis.video.codec`;缺则 ffprobe `codec_name` |
| 起播 | 跨番实测的**中位数** | mpv `playback.timeToPlayingMillis`(**resume→播放位置首次前进**,起播含缓冲;mpv 的 PLAYING 状态同步翻转不含缓冲,故以位置前进为判据) |
| 可播 ✅ | run_eval 快测**或** deep 长播任一实播成功 | 只认 `playback.ran && ok`(`probe_video` 顶层 `ok` 在播放器不可用时退回 HTTP 探测结论,不作数);deep 失败不记 ❌(二次解析可能只是 URL 过期) |

- 码率展示保留**两位小数**(如 `1.46M`),避免四舍五入后看起来达到了阈值。
- 缺数据(从未成功播放)显示 `-`,并且**视为不满足**任何分级阈值要求。
- mpv 原生库没加载 (`mediaAnalysis.available == false`) 时脚本**直接中止评测**——否则 HTTP 可达会被当可播,产出虚假报告。
- 伪装扩展名分片(`.jpeg`/`.png`)需 ffprobe/ffmpeg 加 `-allowed_extensions ALL`(ffmpeg 7.1+ 再加 `-extension_picky 0`,脚本自动探测追加);ffprobe 仍拒绝的流,该线路分辨率/码率退用 mpv 实播数据。

## 3. 广告评判口径

### 判定来源

- **等级只认 agent 真实读图**:每条线路派一个 subagent 读取该线路深采的多点截图,返回结构化判定(`AD_LEVEL` none/low/medium/high、`POSITION`、`SEEN_AT` 出现的采样点、`EVIDENCE` 证据原文、`PERSISTS` 是否全程存在),汇总写入 `combined/frames_verdicts.json`。subagent 的 prompt 附上该线路 `adDetect` 的疑似广告时间段(若有),提示重点看哪些帧。
- `probe_video` 的 `adSuspicion` 等**算法启发式绝不单独作为判定依据**(报告生成器不读它)。注意区分:`suspicion` 是**启发式猜测**,不作数;`hlsFilter.filterable` 是**真实过滤器的执行结果**,仅对"滤不滤得掉"有权威性(见下节)。
- **未判定**的线路显示 `-`/❔,不满足任何"广告上限"要求(即分级最多 T5,排序排最后),不拿启发式充数。
- 判定键可以是源级(`"源名"`)或线路级(`"源名/线路名"`),线路级优先。

### HLS 结构探测与客户端过滤器 (detect_hls_ads)

深采阶段对每条线路调用 MCP `detect_hls_ads`(`probe_video` 的 `adAnalysis` 也带同构的 `hlsFilter` 子结构):拉取 m3u8(master→media)后同时跑**结构启发式**与 **Ani 真实客户端 HLS 广告过滤器(HlsManifestFilter)**:

- `hlsFilter.filterable == true` 表示客户端的自动广告过滤**会在 App 内滤除**这些插入片段;`removedGroups` 给出各段时间范围(秒)与分片数。
- 该结论是**"插入片段滤不滤得掉"的权威依据**:看图确认的插入广告若落在可滤除段内,**不因该片段单独判重**,按滤除后剩余可见广告归级;`filterable == false` 或没有 `hlsFilter` 数据(非 HLS / 清单拉取失败,JSON 会省略 null 字段)→ 默认按滤不掉处理(判重)。
- playlist 分析**看不见烧录进画面的水印/横幅**——广告有没有、什么等级仍以视觉读图为准;结构探测只是辅助证据与加采依据。
- 注意 `status == unchanged` 且 reason 为 `all_segments_candidate` / `encrypted_implicit_iv` / `byterange_implicit_offset` 时,过滤器是**检出了候选但拒绝滤除**(防误删/加密限制),不是"无插入片段"——按滤不掉口径处理,报告行文也须如此区分。
- 报告线路页在"广告判定"下附一行 HLS 结构探测结论(检出几组插入片段、时间范围、可否自动滤除)。

### 等级定义

| 等级 | 定义 |
|---|---|
| 无 (none) | 干净正片 |
| 轻 (low) | 横幅**仅在片头/片尾**出现,正片中没有 |
| 中 (medium) | **片中也会出现**横幅(时有时无,非全程) |
| 重 (high) | **水印/横幅一直在**(全程);或存在**插入的广告视频片段**(整屏广告画面,非正片——前贴片/中插,无论出现在哪里)且 HLS 自动广告过滤滤不掉 |

- 判重不重看**是否全程存在**,不看横幅大小。
- 评测直连播放(mpv)、未经过 Ani 客户端的 HLS 广告过滤;插入片段**滤不滤得掉以 `hlsFilter` 为准**(见上节):可滤除的不因该片段单独判重,`filterable == false` 或无数据默认按滤不掉处理(判重)。
- 采样近似(长播只覆盖开头 28–60s): 广告只在 0–3s 出现、8s 起消失 ≈ 片头(轻);8s 及以后仍间歇出现 ≈ 片中(中);所有采样点都在 ≈ 全程(重)。**拿不准往重的方向判**。

## 4. 线路能力分级 (T0–T6)

每条实测线路按硬指标分级,**从上往下取第一个满足的**。给客户端排优先级用。

| Tier | 含义 | 标准 |
|---|---|---|
| **T0** | 最高优先:该源查询成功,客户端可**直接选择,无需等待其他源** | 无广告 + 1080P + 码率 ≥ 1.8M + 起播 ≤ 3s |
| **T1** | 高优先 | 无广告 + 1080P + 码率 ≥ 1.5M + 起播 ≤ 5s |
| **T2** | 普通 | 无广告 + 1080P + 码率 ≥ 1.2M + 起播 ≤ 5s |
| **T3** | | 1080P + 满足任一:① 轻度及以下广告 + 码率 ≥ 1.5M + 起播 ≤ 8s;② 无广告 + 码率 ≥ 1.0M + 起播 ≤ 8s |
| **T4** | | 轻度及以下广告 + 1080P + 码率 ≥ 1.0M + 起播 ≤ 8s |
| **T5** | | 1080P + 码率 ≥ 1.0M |
| **T6** | 兜底 | 无要求 |

**硬性约束**(日后调整标准时不得破坏):

- **重度广告**线路(全程水印,或滤不掉的插入广告片段)**必须落在 T5 及之后**。
- "无广告 / 轻度广告"只认视觉判定,**未判定不算**。
- 缺码率/起播数据视为不满足该项要求。

## 5. 报告规范

`README.md`(总索引)结构固定:

1. 标题 + `**评测日期: YYYY-MM-DD**` + 方法一句话 + 测试番剧列表;
2. 三个计数:稳定可用 / 部分可用 / 全部失败;
3. `## 🏆 推荐线路` — 每源挑一条最佳线路(无广告 + 有实测数据 + 跨番成功 ≥ n-1 + 画质高 + 码率高 + 起播快),直接告诉用户播哪条;
4. `## 📶 线路能力分级 (Tier)` — **最终 tier 表**,每条实测线路一行。列: `源(线路)`(如 `omofun111(独家蓝光X)`,同源多线路拆多行)· Tier · 分辨率 · 码率 · 编码 · 起播 · 广告 · 每部番 ✅/❌/—。排序: Tier → 可播番数 → 最佳线路排序键;
5. `## 全部可用源` — 一行一源,分辨率/码率/起播列 = 该源"**最佳线路**"实测值。最佳线路 = 实播成功过的线路中按 **①无广告(未判定排最后)②分辨率 ③码率 ④起播** 逐级排序的第一条;
6. `## 各源线路拆解` — 每源一小节,每线路 × 每部番,`⭐` 标记总表所用最佳线路;
7. `## 🔒 搜索阶段失败` / `## ❌ 解析/匹配阶段失败` — 全挂的源按失败阶段归因(验证码/域名失效 vs selector 失配),给维护者修复方向。

另有 `sources/<tier>-<源>.md` 源页与 `channels/<tier>-<源>-<线路>.md` 线路图文页(广告判定含 **HLS 结构探测辅助行** + 播放采样画廊(0/3/8/15/25s 及加采点) + 媒体信息含**码率来源**(ffprobe 实测均值 / ffprobe format / 播放器统计) + 播放性能 + 跨番实播)。

**符号约定**: 广告 `无/轻/中/重`,源级 `*` = 另有更脏线路;列 `-` = 无此项数据;番格 `✅` 可播 / `❌` 解析到但播放失败 / `—` 该番未解析到此线路。失败原因写人话,不暴露裸异常。

## 6. 如何跑一次评测

用 agent 执行 `.agents/skills/datasource-eval/` skill(依赖 animeko 仓库的 `datasource-test-mcp`——本地 HTTP MCP server,Streamable HTTP `POST /mcp`,播放用 mpv 内核,原生库由脚本自动探测 mediamp worktree 或经 `ANI_MPV_NATIVE_DIR` 指定并以 `-Dani.mpv.native.dir` 注入 JVM,加载失败时 probe_video 退化为纯 HTTP 探测;以及系统的 **ffmpeg/ffprobe**,`brew install ffmpeg`),或手动按 skill 文档依次运行 `run_eval.py → deep_sample.py → (agent 看图判广告) → gen_report.py`。深采后可选跑 `deep_sample.py <dir> --backfill-quick`:对快测真实播放成功、但深采二次解析未复现截图的线路,复用快测保存的视频 URL 补跑与主深采同口径的采样(结构预筛 + 加采点 + 长播多点截图 + ffprobe);补跑失败(URL 过期/无帧)的线路保持"未判定",不拿快测的开头两帧充当视觉证据。改了广告判定(`frames_verdicts.json`)或分级标准后,只需重跑 `gen_report.py`,无需重测。
