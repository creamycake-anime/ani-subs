# 数据源评测规范

本文档定义 ani-subs Web (CSS selector) 数据源的**端到端评测标准**:怎么测、指标怎么算、广告怎么判、线路怎么分级、报告长什么样。

- **权威实现**: `.agents/skills/datasource-eval/`(agent skill,含 `scripts/run_eval.py`、`deep_sample.py`、`gen_report.py`)。报告由 `gen_report.py` **确定性生成**,不手写。
- **修改本规范时**,必须同步修改 skill 的 `SKILL.md` 与脚本实现;反之亦然。
- 评测产物在 `reports/<YYYYMMDD>-css-eval/`。
- 注意:本文的能力分级 **T0–T6** 与 `subs/web/t0/`–`t4/` 的**订阅目录分层是两回事**,后者是客户端订阅里源的静态优先级,前者是评测得出的线路实测能力。

## 1. 评测方法

跨 **3–5 部近年完结的热门番剧**(各源收录率高,能区分"源失效"与"没收录"),对每个 Web selector 源:

1. **全流程解析**: `searchSubjects → selectSubjects → searchEpisodes → selectEpisodes → selectMedia → extractVideo`,`all_channels` 模式,**每条线路都解析**(不设条数上限)。
2. **全量实播**: 每条 resolved 线路都用 Animeko 桌面播放器(MPV)真实播放,采集分辨率、码率、编码、起播耗时、可播性。
3. **深度采样**: 对可用源(跨番至少成功 1 次)按**每部番 × 每条线路全覆盖**长播 28s,在 0/3/8/15/25s 各截一帧。
4. **广告判定**: agent **逐张真实读图**(见 §3),覆盖全部实播成功过的线路。
5. **生成报告**: `gen_report.py` 按本规范产出三层报告(总索引 / 源页 / 线路页)。

要求**完整**:每个源 × 每部番 × 每条线路都要真实播放实测,不许抽样。

## 2. 指标口径

所有指标都来自 **MPV 真实播放实测**(快测 4s + 深采长播 28s 两路数据合并),不是源声称的清晰度。

| 指标 | 单线路聚合口径 | 来源 |
|---|---|---|
| 分辨率 | 跨番实测的**众数** | `mediaAnalysis.video` 的 `width x height` |
| 码率 | 跨番实测的**中位数** | `mediaAnalysis.overallBitrate`(容器整体码率),缺则退视频流 `bitrate` |
| 编码 | 跨番实测的**众数** | `mediaAnalysis.video.codec` |
| 起播 | 跨番实测的**中位数** | `playback.timeToPlayingMillis` |
| 可播 ✅ | run_eval 快测**或** deep 长播任一实播成功 | deep 失败不记 ❌(二次解析可能只是 URL 过期) |

- 码率展示保留**两位小数**(如 `1.46M`),避免四舍五入后看起来达到了阈值。
- 缺数据(从未成功播放)显示 `-`,并且**视为不满足**任何分级阈值要求。

## 3. 广告评判口径

### 判定来源

- **只认 agent 真实读图**:每条线路派一个 subagent 读取该线路深采的多点截图,返回结构化判定(`AD_LEVEL` none/low/medium/high、`POSITION`、`SEEN_AT` 出现的采样点、`EVIDENCE` 证据原文、`PERSISTS` 是否全程存在),汇总写入 `combined/frames_verdicts.json`。
- `probe_video` 的 `adSuspicion` 等**算法启发式绝不作为判定依据**(报告生成器不读它)。
- **未判定**的线路显示 `-`/❔,不满足任何"广告上限"要求(即分级最多 T5,排序排最后),不拿启发式充数。
- 判定键可以是源级(`"源名"`)或线路级(`"源名/线路名"`),线路级优先。

### 等级定义

| 等级 | 定义 |
|---|---|
| 无 (none) | 干净正片 |
| 轻 (low) | 横幅**仅在片头/片尾**出现,正片中没有 |
| 中 (medium) | **片中也会出现**横幅(时有时无,非全程) |
| 重 (high) | **水印/横幅一直在**(全程);或存在**插入的广告视频片段**(整屏广告画面,非正片——前贴片/中插,无论出现在哪里)且 HLS 自动广告过滤滤不掉 |

- 判重不重看**是否全程存在**,不看横幅大小。
- 评测用 MPV 直连播放、未经过 Ani 客户端的 HLS 广告过滤,所以截图里看到的插入广告片段**默认按"滤不掉"处理(判重)**,除非已确认过滤器能滤掉该片段。
- 采样近似(28s 长播只覆盖开头): 广告只在 0–3s 出现、8s 起消失 ≈ 片头(轻);8s 及以后仍间歇出现 ≈ 片中(中);所有采样点都在 ≈ 全程(重)。**拿不准往重的方向判**。

## 4. 线路能力分级 (T0–T6)

每条实测线路按硬指标分级,**从上往下取第一个满足的**。给客户端排优先级用。

| Tier | 含义 | 标准 |
|---|---|---|
| **T0** | 最高优先:该源查询成功,客户端可**直接选择,无需等待其他源** | 无广告 + 1080P + 码率 ≥ 1.8M + 起播 ≤ 3s |
| **T1** | 高优先 | 无广告 + 1080P + 码率 ≥ 1.5M + 起播 ≤ 5s |
| **T2** | 普通 | 无广告 + 1080P + 码率 ≥ 1.2M + 起播 ≤ 5s |
| **T3** | | 1080P + 满足任一:① 轻度及以下广告 + 码率 ≥ 1.5M + 起播 ≤ 8s;② 无广告 + 码率 ≥ 1.0M + 起播 ≤ 8s |
| **T4** | | 轻度及以下广告 + 1080P + 码率 ≥ 1.0M + 起播 ≤ 8s |
| **T5** | | 1080P + 码率 ≥ 1.0M |
| **T6** | 兜底 | 无要求 |

**硬性约束**(日后调整标准时不得破坏):

- **重度广告**线路(全程水印,或滤不掉的插入广告片段)**必须落在 T5 及之后**。
- "无广告 / 轻度广告"只认视觉判定,**未判定不算**。
- 缺码率/起播数据视为不满足该项要求。

## 5. 报告规范

`README.md`(总索引)结构固定:

1. 标题 + `**评测日期: YYYY-MM-DD**` + 方法一句话 + 测试番剧列表;
2. 三个计数:稳定可用 / 部分可用 / 全部失败;
3. `## 🏆 推荐线路` — 每源挑一条最佳线路(无广告 + 有实测数据 + 跨番成功 ≥ n-1 + 画质高 + 码率高 + 起播快),直接告诉用户播哪条;
4. `## 📶 线路能力分级 (Tier)` — **最终 tier 表**,每条实测线路一行。列: `源(线路)`(如 `omofun111(独家蓝光X)`,同源多线路拆多行)· Tier · 分辨率 · 码率 · 编码 · 起播 · 广告 · 每部番 ✅/❌/—。排序: Tier → 可播番数 → 最佳线路排序键;
5. `## 全部可用源` — 一行一源,分辨率/码率/起播列 = 该源"**最佳线路**"实测值。最佳线路 = 实播成功过的线路中按 **①无广告(未判定排最后)②分辨率 ③码率 ④起播** 逐级排序的第一条;
6. `## 各源线路拆解` — 每源一小节,每线路 × 每部番,`⭐` 标记总表所用最佳线路;
7. `## 🔒 搜索阶段失败` / `## ❌ 解析/匹配阶段失败` — 全挂的源按失败阶段归因(验证码/域名失效 vs selector 失配),给维护者修复方向。

另有 `sources/<tier>-<源>.md` 源页与 `channels/<tier>-<源>-<线路>.md` 线路图文页(广告判定 + 播放采样画廊 + 媒体信息 + 播放性能 + 跨番实播)。

**符号约定**: 广告 `无/轻/中/重`,源级 `*` = 另有更脏线路;列 `-` = 无此项数据;番格 `✅` 可播 / `❌` 解析到但播放失败 / `—` 该番未解析到此线路。失败原因写人话,不暴露裸异常。

## 6. 如何跑一次评测

用 agent 执行 `.agents/skills/datasource-eval/` skill(依赖 animeko 仓库的 `datasource-test-mcp`),或手动按 skill 文档依次运行 `run_eval.py → deep_sample.py → (agent 看图判广告) → gen_report.py`。改了广告判定(`frames_verdicts.json`)或分级标准后,只需重跑 `gen_report.py`,无需重测。
